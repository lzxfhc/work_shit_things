# 第一部分：高代码托管竞品深度分析

## 一、市场背景与战略意义

### 1.1 为什么"高代码托管"是Agent平台的战略必争之地？

**核心洞察**：Agent平台正在经历从"低代码尝鲜"到"高代码生产"的关键转型期。

根据Gartner预测，到2026年65%的应用开发将由低代码/无代码平台驱动，但**企业级Agent的核心价值创造仍依赖高代码能力**。原因如下：

| 维度 | 低代码适用场景 | 高代码必需场景 |
|------|---------------|---------------|
| **复杂度** | 标准化工作流、简单对话 | 多Agent协作、复杂推理链、自定义工具 |
| **定制化** | 80%通用场景 | 20%核心差异化场景（往往贡献80%业务价值） |
| **性能要求** | 响应时间不敏感 | 毫秒级延迟、高并发、长任务（8小时+） |
| **安全合规** | 公有云托管可接受 | VPC隔离、私有化部署、审计日志 |
| **集成深度** | API简单调用 | 深度嵌入企业系统、自定义协议 |

**市场规模**：全球Agentic AI市场2025年达75.5亿美元，预计2034年达1990亿美元（CAGR 43.84%）。其中**基础设施层（含高代码SDK/Runtime）预计占30-40%份额**，即2025年约22-30亿美元。

## 二、竞品产品形态深度解析

### 2.1 AWS Bedrock AgentCore —— "企业级全家桶"路线

**产品形态**：
```
┌─────────────────────────────────────────────────────────────┐
│                    AgentCore 架构                           │
├─────────────┬─────────────┬─────────────┬─────────────────┤
│   Runtime   │   Gateway   │   Memory    │   Observability │
│  (执行环境)  │  (MCP网关)   │ (短期+长期)  │   (可观测性)     │
├─────────────┴─────────────┴─────────────┴─────────────────┤
│              Browser Tool  │  Code Interpreter             │
│              (浏览器自动化)  │  (代码执行沙箱)                │
├─────────────────────────────────────────────────────────────┤
│         Identity (OAuth身份)  │  Policy (Cedar策略语言)      │
└─────────────────────────────────────────────────────────────┘
```

**核心设计哲学**：
- **框架无关**：支持LangChain、CrewAI、LlamaIndex、Strands等任意开源框架
- **模型无关**：可用Bedrock托管模型，也可接入外部模型
- **模块化计费**：每个组件独立计费，按需组合

**商业模式深度解析**：

| 组件 | 定价 | 战略意图 |
|------|------|----------|
| **Runtime** | $0.0895/vCPU小时 + $0.00945/GB小时 | 基础设施层，走量为主 |
| **Gateway** | $0.005/千次工具调用 | 绑定MCP生态，形成网络效应 |
| **Memory** | $0.25-0.75/千次操作 | 粘性最高，数据留存=客户留存 |
| **Browser** | 按Runtime计费 | 差异化能力，提升客单价 |
| **Code Interpreter** | 按Runtime计费 | 差异化能力，提升客单价 |

**关键洞察**：
1. **免费试用期至2025年9月** —— 典型的"先圈地后收割"策略
2. **VPC + PrivateLink支持** —— 瞄准金融、医疗等合规敏感客户
3. **自然语言转Cedar策略** —— 降低企业治理门槛，这是其他平台没有的

# 第二部分：运行时组件产品建设规划

## 一、Code Interpreter（代码解释器）

### 1.1 产品定义与技术本质

**什么是Code Interpreter？**
- 让Agent能够**编写并执行代码**的安全沙箱环境
- 本质是**"给AI一台虚拟计算机"**，但有严格的安全隔离

**技术架构选型对比**：

| 隔离技术 | 安全级别 | 启动速度 | 资源开销 | 代表产品 |
|----------|----------|----------|----------|----------|
| **Firecracker MicroVM** | 最高（硬件级） | ~150ms | ~5MB/实例 | E2B、AWS Lambda |
| **gVisor** | 高（内核级） | ~100ms | 更低 | Modal、Google Cloud Run |
| **Kata Containers** | 高（轻量VM） | ~300ms | 中等 | Northflank |
| **Docker容器** | 中（进程级） | ~50ms | 最低 | 传统方案（不推荐） |

**千帆建议**：采用**Firecracker MicroVM**方案
- 理由：企业级安全要求高，AWS Lambda验证过的成熟方案
- E2B已获Fortune 500中50%企业采用，技术路线验证充分

### 1.2 商业模式分析

**行业主流定价模式**：

## 二、Browser Use（浏览器自动化）

### 2.1 产品定义与市场机会

**什么是Browser Use？**
- 让Agent能够**像人一样操作浏览器**：点击、输入、滚动、截图、提取信息
- 本质是**"RPA 2.0"**——用AI视觉理解替代脆弱的DOM选择器

**为什么是巨大机会？**

| 数据点 | 来源 |
|--------|------|
| 67%的RPA项目因维护成本失败 | HfS Research |
| AI Browser Agent ROI达8:1（传统RPA仅2:1） | 行业研究 |
| 2025年浏览器自动化市场预计200亿美元+ | Gartner |
| 60-80%成本节省（AI vs 传统自动化） | 企业案例 |

**核心洞察**：浏览器是企业最常用的应用，但也是自动化最难啃的骨头。谁能提供可靠的Browser Use能力，谁就能切入大量企业自动化场景。

## 三、Computer Use（计算机使用）

### 3.1 产品定义与战略判断

**什么是Computer Use？**
- 让Agent能够**操作完整的桌面环境**（不仅是浏览器）
- 包括：操作系统GUI、桌面应用（Word/Excel/SAP等）、文件系统

**与Browser Use的区别**：

| 维度 | Browser Use | Computer Use |
|------|-------------|--------------|
| **范围** | 仅浏览器内 | 完整桌面 |
| **应用** | Web应用 | 任意桌面应用 |
| **复杂度** | 中等 | 高 |
| **成熟度** | 较高（89%准确率） | 中等（38-56%准确率） |
| **商业价值** | 已验证 | 潜力大但需培育 |

**战略判断**：
- **当前成熟度不足**：Anthropic Computer Use在OSWorld仅38.1%，人类72.4%
- **但战略价值巨大**：能操作SAP、用友等桌面ERP = 切入企业核心业务
- **建议策略**：Q1-Q2预研，Q3试点，Q4小规模商用

### 3.2 商业模式分析

**当前市场定价**：
- Anthropic：按Token计费（Computer Use工具增加466-499 token系统开销）
- 无独立定价，融入模型调用成本

**千帆定价建议**：
- **按会话时长**：￥0.3/分钟桌面会话（比Browser Use贵3倍，反映复杂度）
- **按任务计费**：￥5-20/任务（视复杂度）

### 3.3 产品价值与使用场景

**高价值场景**（需培育）：

| 场景 | 描述 | 潜在价值 | 成熟度 |
|------|------|----------|--------|
| **ERP操作自动化** | 自动操作SAP、用友、金蝶 | 巨大（企业核心） | 低 |
| **Office自动化** | Word排版、Excel数据处理、PPT生成 | 大 | 中 |
| **遗留系统集成** | 无API的老旧系统自动化 | 大 | 低 |
| **软件测试** | 桌面应用自动化测试 | 中 | 中 |

### 3.4 Computer Use Q1-Q4建设规划

| 分维度 | Q1 | Q2 | Q3 | Q4 |
|--------|-----|-----|-----|-----|
| **目标** | 技术预研+方案设计 | 原型开发+内部验证 | Alpha版本+种子客户 | Beta版本+小规模商用 |
| **效果** | 完成技术选型报告 | 内部Demo可演示 | 3-5个种子客户试用 | 10+付费客户 |
| **功能** | - | Linux桌面基础操作 | +Windows桌面/Office套件/截图理解 | +多应用协作/复杂工作流 |
| **性能** | - | 简单任务成功率>60% | 标准任务成功率>70% | 标准任务成功率>80% |

## 四、Mobile Use（移动端使用）

### 4.1 产品定义与市场现状

**什么是Mobile Use？**
- 让Agent能够**操作Android/iOS移动设备**
- 应用场景：App自动化测试、移动端数据采集、跨App工作流

**市场成熟度评估**：

| 框架 | 成功率 | GitHub Stars | 特点 |
|------|--------|--------------|------|
| **DroidRun** | 43%（最高） | 3.8k | 最快增长，原生Android |
| **Mobile-Use** | 接近DroidRun | 增长中 | 支持iOS+Android |
| **Mobile-Agent-v3** | 未公开 | - | 阿里研究项目，NeurIPS 2025 |

**核心挑战**：
- 成功率仅43%，远低于Browser Use的89%
- iOS限制多，自动化难度大
- 企业需求尚未爆发

### 4.2 战略建议：**观望为主，小步试探**

**理由**：
1. 技术成熟度不足（43% vs 89%）
2. 企业移动端自动化需求尚未规模化
3. 投入产出比低于Browser Use和Code Interpreter
